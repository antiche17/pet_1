🚀 1. Что такое CI/CD
CI (Continuous Integration) — автоматическая сборка и проверка кода при каждом изменении.

CD (Continuous Delivery/Deployment) — автоматическая доставка приложения на тестовые/боевые окружения.

Формула:  
CI = проверка → CD = доставка.

🧹 Что такое Static Analysis (статический анализ)
Статический анализ — это автоматическая проверка кода без его запуска.
Инструменты анализируют исходный код и ищут проблемы заранее.

Это как проверка текста на ошибки перед тем, как его прочитать вслух.

🧰 Что такое линтеры
Линтер (linter) — это инструмент, который проверяет стиль и качество кода.

Он ищет:

неправильное форматирование,

неиспользуемые переменные,

потенциальные ошибки,

нарушение код‑стайла,

подозрительные конструкции.

Примеры линтеров:

ESLint (JavaScript)

Pylint/Flake8 (Python)

Checkstyle (Java)

Зачем тестировщику:  
Понимать, что если линтер нашёл ошибку — пайплайн может упасть ещё до тестов.

🧠 Что такое SonarQube
SonarQube — это более продвинутый инструмент статического анализа.
Он проверяет код глубже, чем линтеры.

SonarQube ищет:

баги,

уязвимости,

дублирование кода,

проблемы с архитектурой,

"code smells" — подозрительные места, которые могут привести к ошибкам.

Он формирует отчёт и выставляет Quality Gate — набор правил, которые код должен пройти.

Если Quality Gate не пройден → пайплайн становится красным.

🧩 Как это объяснить на собеседовании
Готовый ответ:

«Static analysis — это автоматическая проверка качества кода без его запуска.
Линтеры проверяют стиль и простые ошибки, а SonarQube анализирует код глубже: ищет баги, уязвимости, дублирование и проблемы качества.
Если статический анализ не пройден, пайплайн останавливается ещё до запуска тестов».

🧪 Что такое мокирование (mocking)
Мокирование — это подмена реальных зависимостей в тестах на фейковые объекты (mocks), которые ведут себя предсказуемо.

Проще говоря:

Мы создаём «поддельные» версии сервисов, API, БД, очередей, чтобы тестировать систему изолированно и стабильно.

🎯 Зачем мокирование нужно в CI
CI‑среда должна быть:

быстрой — тесты не должны ждать реальных сервисов;

стабильной — внешние системы могут быть недоступны;

изолированной — тесты не должны ломать данные в реальной БД;

предсказуемой — одинаковый результат при каждом запуске.

Поэтому мокирование — ключевой инструмент для стабильных автотестов в CI.

🧩 Где используется мокирование
1. API тесты
Вместо реального внешнего сервиса → мок‑сервер.
Например:

платежная система,

сервис авторизации,

сторонний геосервис.

2. Интеграционные тесты
Если один модуль зависит от другого, но второй ещё не готов → мок.

3. UI тесты
UI‑тесты могут мокировать API, чтобы:

не ждать долгих ответов,

не зависеть от нестабильного бэкенда,

тестировать редкие сценарии (ошибки, timeouts).

4. Тесты в пайплайне
Моки позволяют запускать тесты без реальных окружений, что ускоряет CI.

🛠 Примеры инструментов мокирования
Тебе не нужно уметь ими пользоваться, но стоит знать названия:

WireMock — мокирование HTTP API

MockServer — универсальный мок‑сервер

Mockito (Java)

pytest-mock (Python)

Nock (JavaScript)

Cypress fixtures — моки в UI‑тестах

Postman Mock Server

Готовый ответ:

В CI это важно, потому что тесты должны быть быстрыми, стабильными и не зависеть от внешних сервисов.
Моки позволяют воспроизводить любые сценарии, включая ошибки, и обеспечивают предсказуемость пайплайна».

🧩 4. Page Object Model (POM)
Пример base_page.py:

python
class BasePage:
    def __init__(self, page):
        self.page = page

    def click(self, locator):
        self.page.locator(locator).click()

    def fill(self, locator, text):
        self.page.locator(locator).fill(text)

    def get_text(self, locator):
        return self.page.locator(locator).inner_text()
Пример login_page.py:

python
from pages.base_page import BasePage

class LoginPage(BasePage):
    username = "#user-name"
    password = "#password"
    login_btn = "#login-button"
    error_msg = "h3[data-test='error']"

    def login(self, user, pwd):
        self.fill(self.username, user)
        self.fill(self.password, pwd)
        self.click(self.login_btn)

    def get_error(self):
        return self.get_text(self.error_msg)
🧪 5. Пример теста (pytest + Playwright)

def test_success_login(page):

🔄 7. CI (GitHub Actions)
Минимальный пайплайн:

Код
name: UI Tests

on: [push, pull_request]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Install Python
        uses: actions/setup-python@v4
        with:
          python-version: '3.10'
      - name: Install dependencies
        run: pip install -r requirements.txt
      - name: Install Playwright browsers
        run: playwright install
      - name: Run tests
        run: pytest --alluredir=allure-results
      - name: Upload Allure results
        uses: actions/upload-artifact@v3
        with:
          name: allure-results
          path: allure-results
